# Client Onboarding Checklist

## When to Use

- User has signed a new freelance or consulting client and needs to set up the engagement
- User wants to create a standardized client onboarding process
- User asks how to kick off a new client relationship with clear expectations
- User needs a checklist for what to cover before starting project work
- User wants to avoid scope creep and miscommunication at the start of an engagement
- Do NOT use when: user is onboarding a new employee (use HR skills), user needs an internal project kickoff (use project management skills), user wants to write a client proposal (use `freelance-proposal-writer`)

## Process

1. **Gather engagement context.** Collect the details of the new client:
   - Client name, company, and primary contact person
   - Project type and scope (from the signed proposal or agreement)
   - Contract status (signed/pending)
   - Agreed timeline and key milestones
   - Communication preferences (email, Slack, phone, meeting frequency)
   - Billing structure (hourly, project-based, retainer, payment terms)
   - What the client needs to provide (assets, access, content, feedback)

2. **Build the pre-kickoff checklist.** Before the first meeting:
   - Contract signed and countersigned
   - Initial payment received (if required before work begins)
   - Client contact information documented (primary contact, billing contact, technical contact if different)
   - Project management or collaboration tools set up (shared folder, project board, communication channel)
   - Internal project brief created (user's own notes on scope, risks, timeline)
   - Welcome email sent with kickoff meeting details and agenda

3. **Create the discovery call agenda.** Structure the first conversation:
   - Introductions and roles (who does what on both sides)
   - Project goals review (confirm understanding of objectives from the proposal)
   - Success criteria definition (how will both parties know the project succeeded?)
   - Brand guidelines, style preferences, or technical requirements review
   - Content and asset inventory (what the client has vs. what needs to be created)
   - Communication plan (preferred channels, response time expectations, meeting cadence)
   - Timeline walkthrough (key dates, milestones, review points)
   - Questions and concerns from either side
   - Next steps and action items with owners and dates

4. **Build the kickoff document.** Create a shared reference:
   - Project summary (1-2 paragraphs restating scope and goals)
   - Team directory (names, roles, contact information for both sides)
   - Communication plan (channels, frequency, escalation process)
   - Timeline with milestones and review dates
   - Client responsibilities and deadlines for their deliverables
   - Revision process (how feedback is submitted, how many rounds)
   - Change request process (how scope changes are handled)
   - File sharing and access information

5. **Set communication expectations.** Define the operating rhythm:
   - Status update frequency (weekly email, bi-weekly call, etc.)
   - Response time expectations (business hours, same-day, next-day)
   - Preferred feedback format (in-document comments, email, meeting)
   - Meeting cadence and duration
   - Escalation path (who to contact if something goes wrong)
   - Out-of-scope request handling (how to raise additional needs)

6. **Define the first deliverable milestone.** Set the initial target:
   - What will be delivered first and when
   - What feedback is needed from the client and by when
   - How the first deliverable review will be conducted
   - Success criteria for the first milestone
   - Adjustment point: review process and communication after the first deliverable to calibrate expectations

## Rules

1. Always produce a complete onboarding document with checklists, agenda, and shared reference -- never generic advice about client management
2. Every checklist item must have a checkbox (- [ ]) for the user to track completion
3. The discovery call agenda must include specific time allocations for each section -- vague agendas waste client time
4. Client responsibilities must include specific due dates, not vague references to "as soon as possible"
5. Include a change request process in every onboarding document -- this is the primary defense against scope creep
6. Communication expectations must include response time commitments from BOTH sides, not just the user
7. Include a process check after the first deliverable -- this is the adjustment point for the entire engagement
8. NEVER assume the client knows how the user's process works -- explain everything, even things that seem obvious
9. The revision process must specify the number of included rounds and the cost of additional rounds
10. The kickoff document becomes the single source of truth for the engagement -- everything agreed upon should be in this document

## Edge Cases

- **Client is a repeat engagement (returning client):** Simplify the onboarding by referencing the previous engagement. Skip introductions and basic process review. Focus on what is different about this project: new scope, updated timeline, any process changes from last time.
- **Multiple stakeholders on the client side:** Add a RACI (Responsible, Accountable, Consulted, Informed) matrix to the kickoff document. Identify who provides feedback, who approves deliverables, and who makes final decisions. This prevents conflicting feedback from multiple client contacts.
- **Very small project (under 1 week):** Collapse the onboarding to a single email. Include: confirmed scope, delivery date, what you need from the client, payment terms. Skip the formal discovery call -- a 15-minute alignment call or email exchange is sufficient.
- **Client has no process experience (first time hiring a freelancer):** Add extra explanation to every section. Define terms like "revision round," "scope change," and "milestone." Include a brief "how this works" section at the top of the onboarding document that explains the typical freelance engagement flow.
- **Remote client in a different time zone:** Add time zone information to every meeting reference. Identify a shared working hours window (e.g., "Both parties are available 10 AM - 2 PM EST"). Emphasize async communication with clear written documentation over real-time meetings.

## Example

**Input:** "I just signed a freelance UX design project with a startup called TaskFlow. The project is redesigning their onboarding flow for new users. The client contact is Jamie Park, Head of Product. The project is 4 weeks, $8,000 fixed fee (50/50 payment split). I include 2 rounds of revisions. We'll communicate via Slack and have weekly check-in calls."

**Output:**

## Client Onboarding Document: TaskFlow - Onboarding Flow Redesign

**Date:** [Current date]
**Client:** Jamie Park, TaskFlow
**Primary contact:** Jamie Park, [email], [phone]
**Project:** Redesign of new user onboarding flow
**Timeline:** [Start date] to [Start date + 4 weeks]

---

### Pre-Kickoff Checklist

- [ ] Contract signed by both parties
- [ ] Initial payment of $4,000 (50%) received
- [ ] Client contact information documented
  - Primary: Jamie Park, [email], [phone]
  - Billing: [Name if different], [email]
- [ ] Shared project folder created (Google Drive or Figma workspace)
- [ ] Slack channel created and Jamie invited
- [ ] Welcome email sent with kickoff meeting details and agenda
- [ ] Internal project brief completed (current onboarding flow review, competitive analysis notes)

### Discovery Call Agenda

**Date:** [Scheduled date and time]
**Duration:** 45 minutes
**Attendees:** [User name], Jamie Park

1. **Introductions and roles** (5 min)
   - Who approves designs? Jamie or another stakeholder?
   - Who provides technical constraints? Engineering contact needed?

2. **Project goals review** (10 min)
   - What is the primary goal of the onboarding redesign? (Activation rate, time to value, drop-off reduction?)
   - What metrics define success? (e.g., "Reduce onboarding drop-off from 40% to 20%")
   - What is working in the current onboarding? What is not?

3. **Requirements and preferences** (15 min)
   - Existing design system or component library?
   - Brand guidelines, typography, color palette
   - Technical constraints (what can engineering build in the current stack?)
   - User research or analytics data available (where do users drop off today?)

4. **Communication plan** (5 min)
   - Slack for day-to-day, weekly video call for milestone reviews
   - Feedback turnaround: 2 business days after each deliverable
   - Jamie is the single point of feedback (consolidated, not multiple reviewers)

5. **Timeline walkthrough** (5 min)
   - Week 1: Research and wireframes
   - Week 2-3: High-fidelity design
   - Week 4: Revisions and final delivery

6. **Questions and next steps** (5 min)
   - Jamie to share current onboarding flow screenshots and analytics by Day 2
   - [User] to deliver wireframes by end of Week 1
